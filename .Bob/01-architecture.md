# Architecture

## Overview

The operator follows the standard controller-runtime pattern: it watches `KubeInvaders` custom resources and reconciles the cluster to match. One `KubeInvaders` CR equals one running instance of the game.

```
KubeInvaders CR (game.kubeinvaders.io/v1alpha1, namespaced)
        │
        ▼
kubeinvaders-operator (controller-manager Deployment)
        │ reconciles
        ├── Deployment       (the game: OpenResty + Redis + Lua, port 8080)
        ├── Service          (ClusterIP/NodePort/LoadBalancer → port 8080)
        ├── Ingress          (optional, spec.ingress.enabled)
        ├── ServiceAccount   (mounted into the game pod)
        ├── ClusterRole      (chaos permissions, named kubeinvaders-<ns>-<name>)
        └── ClusterRoleBinding
```

## Reconciliation

`KubeInvadersReconciler.Reconcile` runs on every CR change and on changes to owned resources (`Owns()` watches on Deployment, Service, ServiceAccount, Ingress):

1. Fetch the CR; exit if deleted.
2. Finalizer handling — cluster-scoped ClusterRole/ClusterRoleBinding cannot have a namespaced owner reference, so deletion is handled via the `game.kubeinvaders.io/finalizer` finalizer: on CR delete, the controller removes both, then removes the finalizer.
3. CreateOrUpdate each resource (idempotent server-side reconcile): ServiceAccount → ClusterRole/Binding → Deployment → Service → Ingress (deleted if `ingress.enabled: false`).
4. Status update: `readyReplicas` mirrored from the Deployment; an `Available` condition flips true when ≥1 replica is ready.

Namespaced resources carry owner references to the CR, so Kubernetes garbage-collects them automatically on CR deletion.

## Environment variables passed to the game

| Env | Source |
|---|---|
| `NAMESPACE` | `spec.targetNamespaces` joined with `,` |
| `APPLICATION_URL` | `spec.applicationURL`, else ingress host, else `<name>.<ns>.svc.cluster.local:8080` |
| `REDIS_HOST` | `127.0.0.1` (Redis runs inside the game container) |
| `TOKEN` | read by the game from the mounted ServiceAccount token (not set by operator) |
| anything else | `spec.extraEnv` (ALIENPROXIMITY, HITSLIMIT, UPDATETIME, DISABLE_TLS, …) |

## RBAC model (two layers)

**Layer 1 — what the operator grants the game.** Each CR gets a ClusterRole bound to its ServiceAccount: delete on `pods`/`pods/log`, full CRUD on `jobs` (batch/extensions), cluster-wide `get/list/watch` on everything, and `update` on `subresources.kubevirt.io/virtualmachines/restart`. This mirrors the game's needs: shooting an alien deletes a pod; the UI lists pods/nodes across target namespaces.

**Layer 2 — what the operator itself holds.** Kubernetes refuses RBAC grants the granter doesn't hold (privilege-escalation prevention), so the operator's ClusterRole (`config/rbac/role.yaml`, mirrored into the CSV `clusterPermissions`) contains everything from layer 1 plus management permissions for deployments, services, serviceaccounts, ingresses, clusterroles/bindings, and the `kubeinvaders` CRD group. RBAC rules are declared as `+kubebuilder:rbac` markers on the controller and regenerated with `make manifests`.

## Design decisions

- **Group/version**: `game.kubeinvaders.io/v1alpha1`; plural forced to `kubeinvaders` (the auto-pluralizer mangles it).
- **Cluster-unique RBAC names**: `kubeinvaders-<namespace>-<name>` so multiple CRs in different namespaces don't collide.
- **Deployment selector** is set once at creation and never mutated (selectors are immutable).
- **CSV UI metadata** lives as `+operator-sdk:csv:customresourcedefinitions:` markers in the Go types, so `make bundle` regenerates it instead of losing it.
- The game image itself (`luckysideburn/kubeinvaders`) is unmodified upstream; the operator only orchestrates it.
