# Shipping & Releases

## The release command

```bash
cd operator
./build.sh build --new-version --ship
```

This: bumps the patch version everywhere it is pinned, runs all quality gates, then builds and pushes three multi-arch images to Docker Hub in dependency order:

1. `fumbles/kubeinvaders-operator:v<V>` — `make docker-buildx` (cross-compiled, linux/amd64+arm64)
2. `fumbles/kubeinvaders-operator-bundle:v<V>` — `make bundle-buildx`
3. `fumbles/kubeinvaders-operator-catalog:v<V>` — `make catalog-buildx` (renders the catalog **from the just-pushed bundle**, so order matters)

`--ship` warns if the git tree is dirty. Afterwards, commit and tag:

```bash
git add -A && git commit -m "operator v<V>" && git tag operator-v<V>
```

## Where the version is pinned

`--new-version` (and an explicit `./build.sh build X.Y.Z`) keeps these in sync:

| File | Field |
|---|---|
| `Makefile` | `VERSION ?=` |
| `config/manager/kustomization.yaml` | `newTag` (operator image deployed by the CSV) |
| `config/manifests/bases/...clusterserviceversion.yaml` | `containerImage` annotation + `replaces` |
| `config/catalog/catalogsource.yaml` | catalog image tag |
| `bundle/` | regenerated by the bundle gate with the new version |

## OLM upgrade graph

Each bump sets `spec.replaces: kubeinvaders-operator.v<previous>` in the CSV base, and `make catalog` copies that into the channel entry. OLM uses this chain to upgrade installed operators automatically (Subscriptions with `installPlanApproval: Automatic` upgrade as soon as the CatalogSource re-polls — every 10 minutes).

If the previous version was never actually published to a catalog, delete the `replaces:` line before shipping, or OLM will look for a CSV that never existed.

Note the current single-bundle catalog only contains the newest bundle. The upgrade edge still works (dangling `replaces` is legal), but if you ever need multiple versions installable side-by-side, render multiple bundles into the catalog in `make catalog`.

## Rollback / hotfix

Images are immutable per tag. To roll a cluster back: point the Subscription at the older CSV (`startingCSV`) or push a new patch version whose `replaces` chain skips the bad release. Don't retag images.
